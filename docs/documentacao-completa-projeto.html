1. Visão geral do projeto

O projeto é uma interface de consulta sobre escolas rurais do Brasil. Na prática, ele faz três coisas ao mesmo tempo:

  1. carrega uma base principal local com dezenas de milhares de escolas;
  2. permite filtrar, buscar, paginar e abrir fichas detalhadas de cada escola;
  3. enriquece a leitura com indicadores externos do IBGE e com um cache complementar por escola.

Isso significa que o projeto não é apenas uma página estática. Ele é uma pequena aplicação front-end, toda escrita em JavaScript puro, com estado global, renderização dinâmica, integrações assíncronas e comportamento responsivo para desktop e mobile.

Camada 1 - Base local

O arquivo escolas-rurais.json é a espinha dorsal do sistema. Ele alimenta os filtros, os contadores, a tabela de resultados e a maior parte da ficha da escola.

Camada 2 - Indicadores territoriais

O projeto consulta APIs do IBGE para buscar alfabetização, renda per capita, escolaridade por território e os metadados oficiais de estados e municípios.

Camada 3 - Enriquecimento por escola

O arquivo school-enrichment-cache.json guarda uma camada extra derivada de uma ficha pública por escola. Esse cache é consumido pela ficha do modal.

Ponto central

A ideia arquitetural mais importante do projeto é esta: a base local controla a navegação principal, e as fontes externas entram como enriquecimento progressivo. Isso deixa a interface rápida para abrir, mesmo quando algumas integrações demoram ou falham.

Origem da base local

O arquivo escolas-rurais.json não surgiu do nada. Ele é o resultado do trabalho de estruturação feito em cima da planilha escolas-rurais-atualizadas, fornecida pelo professor Átila Barros. Em outras palavras: a planilha foi a matéria-prima e o JSON foi a forma final preparada para a aplicação.

2. Como estudar este código sem se perder

Quem está começando costuma tentar entender tudo de uma vez. Isso gera ansiedade e confusão. Neste projeto, a melhor estratégia é seguir uma ordem de leitura.

  1. Abra index.html e veja apenas a estrutura da página.
  2. Depois leia style.css só para identificar os grandes blocos visuais.
  3. Em script.js, comece pelo topo: constantes, objeto app e objeto els.
  4. Em seguida, leia o ciclo principal: bindEvents(), loadLocalData(), applyFilters(), renderTable().
  5. Só depois disso avance para os painéis do IBGE e para a ficha da escola.
Regra prática

Quando olhar para uma função, faça sempre estas três perguntas:

  1. de onde ela recebe informação;
  2. o que ela transforma;
  3. qual parte da tela ela atualiza.

3. Mapa dos arquivos

O projeto principal está concentrado em poucos arquivos. Isso tem a vantagem de ser fácil de abrir e a desvantagem de concentrar muita responsabilidade em um único script.js.

Arquivo Papel O que você costuma editar aqui
index.html Estrutura da página, filtros, cards, áreas de resultado e containers dos modais. Adicionar novas seções, novos botões, novos elementos que depois serão controlados pelo JS.
style.css Visual da home, tabela, dashboard, modais, ficha da escola, responsividade e impressão visual. Ajustar layout, tipografia, espaçamento, responsividade e estados visuais.
script.js Lógica da aplicação: estado global, fetch, filtros, renderização, integração com APIs e modais. Quase toda manutenção funcional do projeto.

Importante

O projeto roda só com front-end e arquivos locais. Isso é ótimo para simplicidade, mas significa que parte das integrações depende de o navegador permitir requisições e de o arquivo ser aberto em contexto compatível. Por isso existe fallback com XMLHttpRequest no carregamento da base local.

4. Fluxo geral da aplicação

Antes de entrar em cada arquivo, vale visualizar o ciclo completo. Este é o fluxo real da aplicação:

1. O navegador abre index.html 2. O HTML carrega style.css e script.js 3. O script cria constantes, estado global e referências DOM 4. bindEvents() registra os eventos de clique, change, keydown e input 5. loadLocalData() carrega escolas-rurais.json 6. detectColumns() descobre quais colunas do JSON correspondem a cada campo lógico 7. buildFilters() preenche os selects 8. applyFilters() gera app.filteredRows 9. renderStats() e renderTable() desenham a tela principal 10. updateLiteracyPanel(), updateIncomePanel() e updateEducationDashboard() consultam o IBGE 11. Ao clicar numa linha, openSchoolModalById() monta a ficha da escola 12. A ficha carrega complementos: enriquecimento local, renda e alfabetização

Repare que o fluxo é incremental. A home funciona primeiro com a base local. Depois, em paralelo, os indicadores externos completam a experiência. Isso evita que a tela fique inutilizável enquanto uma API externa responde.

5. Anatomia do HTML

O index.html é curto e organizado por seções. Isso é bom porque separa bem os containers visuais. O HTML não tenta fazer lógica. Ele apenas cria “lugares” para o JavaScript preencher.

5.1 Por que quase tudo tem id?

Porque o JavaScript precisa encontrar esses elementos rapidamente com document.getElementById(). Esse padrão aparece no objeto els. A lógica é simples: o HTML define o ponto de montagem, e o JS centraliza as referências a esses pontos num único lugar.

5.2 O HTML já vem com dados prontos?

Não. A maior parte do conteúdo dinâmico começa com textos como “Carregando...” ou containers vazios. Isso é intencional. Depois que os dados chegam, o JavaScript substitui o conteúdo.

6. Anatomia do CSS

O style.css concentra todo o comportamento visual. Ele não está separado por comentários grandes, então a melhor maneira de lê-lo é por blocos de classe.

Parte 1 - Base visual

No topo do arquivo ficam as variáveis CSS em :root, as regras globais, o fundo geral, o padrão dos cards e os estilos de formulário.

Parte 2 - Home

Depois vêm filtros, stats, painéis, tabela e dashboard. Essa parte define praticamente toda a home.

Parte 3 - Ficha da escola

Mais adiante aparecem as classes do modal, da estrutura interna da ficha, do mapa e dos blocos complementares.

Parte 4 - Responsividade

No final estão os media queries para desktop menor, tablet, smartphone e ajustes específicos do modo mobile.

6.1 Decisão importante: lista mobile em vez de tabela completa

No desktop, a área de resultados é uma tabela com várias colunas. No mobile, isso foi transformado numa lista em que apenas a coluna da escola permanece visível. Essa decisão reduz ruído e faz sentido para toque: no celular a pessoa toca no nome e abre a ficha para ver o restante.

6.2 Onde mexer quando o layout quebra

  1. procure a classe visual do bloco em si;
  2. depois procure o media query que altera esse bloco;
  3. veja se existe alguma regra com max-width, grid-template-columns ou display alterando o comportamento.
Regra prática

Em problemas de layout deste projeto, os culpados mais comuns são: max-width, grid-column, white-space, word-break e regras diferentes dentro dos media queries.

7. Arquitetura do JavaScript

O script.js é o coração da aplicação. Ele é grande e faz muita coisa, então o jeito certo de entendê-lo é dividi-lo mentalmente em camadas.

7.1 Camada de configuração

No topo do arquivo ficam constantes como DATA_URL, IDs de tabelas do SIDRA, nomes de fonte, chaves de filtros e a definição das colunas da tabela principal. Essa parte responde à pergunta: “quais dados o sistema conhece?”.

7.2 Camada de estado global

O objeto app é o estado global do projeto. Ele guarda:

  • rawRows: todas as linhas da base principal;
  • filteredRows: linhas depois da aplicação dos filtros;
  • columns: mapeamento entre campo lógico e nome real da coluna do JSON;
  • estado de paginação;
  • estado dos modais;
  • caches de alfabetização, renda, escolaridade e enriquecimento por escola.
Conceito importante

Quando falamos que um dado é “global” neste projeto, normalmente estamos dizendo que ele fica dentro do objeto app e pode ser reutilizado por várias funções. Esse foi justamente o mecanismo escolhido para resolver a necessidade de compartilhar dados do fetch com o resto da aplicação.

7.3 Camada de referências DOM

O objeto els centraliza as referências aos elementos do HTML. Isso evita ficar repetindo document.getElementById() pelo arquivo inteiro e deixa mais claro quais nós da interface existem.

7.4 Camada utilitária

Funções como normalize(), text(), escapeHtml() e renderValue() fazem pequenos tratamentos que são reutilizados o tempo todo. Elas resolvem problemas clássicos:

  • comparar texto sem acento e sem diferença entre maiúsculas e minúsculas;
  • garantir que valores null ou vazios não quebrem a interface;
  • evitar injeção de HTML ao montar strings para innerHTML.

7.5 Camada de acesso ao dado local

Aqui entram detectColumn(), detectColumns() e get().

A função detectColumns() olha o cabeçalho real do JSON e tenta descobrir qual coluna corresponde a cada campo lógico do sistema. Isso é muito útil quando a planilha de origem muda ligeiramente de nome, por exemplo “Município” versus “Municipio”.

Depois que esse mapeamento é criado, a função get(row, key) permite acessar um campo por nome lógico, sem que o resto do projeto precise saber qual era o texto exato da coluna original.

8. Modelo de dados local

A base principal do projeto é o arquivo escolas-rurais.json. O código não acessa esse JSON diretamente pelo nome cru das colunas. Em vez disso, ele mapeia os campos esperados e depois trabalha com chaves lógicas.

8.1 Campos principais esperados

Chave lógica Significado Onde aparece
schoolNome da escolaTabela, busca, ficha do modal
inepCódigo INEPTabela e identificação da ficha
ufSigla do estadoFiltros, territorialização, fichas e cruzamentos IBGE
cityMunicípioFiltros, tabela, indicadores municipais
dependencyDependência administrativaFiltro “Categoria” e tags da ficha
differentiatedLocationÁrea diferenciadaFiltro “Área” e ficha

8.2 O que é um campo derivado

Um campo derivado é um campo que não existe de forma explícita no JSON, mas pode ser calculado a partir de outro. O melhor exemplo aqui é a região. O JSON traz a UF. A região é obtida consultando o objeto regioesPorUF.

É por isso que o filtro de região não depende de uma coluna real da planilha. Ele depende da função getFilterValue().

9. Sistema de filtros

O sistema de filtros é uma das partes mais importantes do projeto. Ele foi desenhado para ter um núcleo genérico, mas também aceita exceções, como filtros derivados.

9.1 Estrutura-base dos filtros

O array FILTER_SELECTS lista os filtros do tipo select. Cada item informa:

  • qual elemento do DOM representa o filtro;
  • qual chave lógica do dado ele filtra;
  • qual rótulo deve aparecer quando nada foi selecionado.

9.2 Filtros que se montam automaticamente

A função buildFilters() percorre esse array e preenche os selects com valores únicos extraídos da base. Essa é a razão pela qual adicionar um filtro simples é relativamente barato: se o campo já existe no JSON, o próprio sistema consegue montar as opções.

9.3 Filtro derivado de região

A região não vem pronta na base. Então o projeto usa getFilterValue(row, key). Se a chave pedida for region, a função consulta getRegionByUf() em vez de procurar uma coluna.

9.4 Encadeamento entre região, UF e município

  1. Ao escolher uma região, o filtro de UF é reconstruído para exibir apenas estados daquela região.
  2. Ao escolher uma UF, o filtro de região é sincronizado automaticamente e o filtro de município é reconstruído com as cidades daquela UF.
  3. Ao tentar abrir município sem UF, a interface bloqueia e mostra um aviso leve.
Região muda -> limpa UF -> limpa Município -> syncUfFilterOptions() -> syncCityFilterOptions() -> applyFilters() UF muda -> sincroniza Região -> syncUfFilterOptions() -> syncCityFilterOptions() -> applyFilters()

9.5 Busca por texto

A busca por nome é combinada com os filtros. Isso quer dizer que a aplicação não escolhe entre “buscar” ou “filtrar”. Ela faz as duas coisas juntas dentro de applyFilters().

9.6 Por que existe debounce()?

Porque, sem ele, cada letra digitada dispararia imediatamente uma filtragem completa sobre a base inteira. O debounce espera alguns milissegundos após a digitação parar. Isso melhora a experiência.

10. Busca, paginação e tabela

A sequência principal da home é esta:

  1. applyFilters() recalcula app.filteredRows;
  2. refreshResults() chama renderStats() e renderTable();
  3. renderTable() monta a tabela atual com base na página visível.

10.1 Paginação

A paginação é simples e eficiente. O sistema não pede novas páginas ao servidor. Ele já tem tudo em memória e só calcula um recorte da lista filtrada: 

const startIndex = (app.currentPage - 1) * app.pageSize;
const endIndex = Math.min(startIndex + app.pageSize, app.filteredRows.length);
const visibleRows = app.filteredRows.slice(startIndex, endIndex);

Isso é bom porque a base é local. Não existe paginação remota aqui.

10.2 A tabela é realmente clicável por linha

Em vez de colocar links espalhados, o projeto marca cada linha com data-school-id e usa delegação de evento. Isso permite abrir a ficha clicando em qualquer ponto da linha.

10.3 Desktop e mobile têm duas leituras diferentes

No desktop, a tabela aparece com várias colunas. No mobile, quase todas as células são escondidas e fica visível apenas a célula da escola, transformada em um cartão enxuto. Esse é um bom exemplo de responsividade orientada à tarefa, não apenas à largura.

11. Painéis e indicadores externos

O projeto consome APIs do IBGE em três frentes distintas:

  • alfabetização;
  • renda per capita;
  • dashboard de escolaridade por território.

11.1 Duas APIs diferentes do IBGE

APIPapel no projeto
IBGE LocalidadesResolver estados, regiões e municípios oficiais, com seus códigos.
SIDRATrazer os indicadores estatísticos em si.

11.2 Por que o projeto guarda caches em Map

Porque alguns indicadores são consultados por território várias vezes. Se cada mudança de filtro disparasse tudo do zero, a interface ficaria lenta e faria requisições repetidas sem necessidade.

Por isso o estado global guarda, por exemplo:

  • app.literacy.regionRates e app.literacy.stateRates;
  • app.income.regionValues e app.income.stateValues;
  • municipalityCatalogByUf e municipalityValuesByUf;
  • promises pendentes para evitar requisições duplicadas.

11.3 Painel de alfabetização

A função ensureBaseLiteracyLoaded() carrega Brasil, regiões e estados. Quando há uma UF selecionada, ensureMunicipalityLiteracyByUf() completa os dados dos municípios daquela UF.

Depois disso, buildLiteracySnapshot() monta um retrato do contexto atual, e createLiteracyMetrics() transforma esse retrato em cards renderizáveis.

11.4 Card de renda per capita

A renda per capita da home mostra o recorte mais relevante do momento:

  • sem território selecionado: Brasil;
  • com região selecionada: região;
  • com UF selecionada: UF;
  • com município selecionado: ainda usa a UF, porque esse foi o recorte definido para a home.

11.5 Dashboard de desigualdade educacional

O dashboard territorial usa perfis de escolaridade com base na população de 25 anos ou mais. Ele trabalha com indicadores cumulativos:

  • fundamental incompleto;
  • pelo menos fundamental completo;
  • pelo menos ensino médio completo;
  • superior completo;
  • anos médios de estudo.
Interpretação correta

Como parte dos indicadores é cumulativa, a soma pode passar de 100%. Isso não é bug. Isso acontece porque quem concluiu o ensino superior já está incluído em “pelo menos ensino médio completo” e também em “pelo menos fundamental completo”.

12. Ficha da escola e modais

A ficha da escola é montada inteiramente por JavaScript. Ela não existe pronta no HTML. Quando o usuário clica numa linha, o sistema chama openSchoolModalById(), injeta o HTML da ficha e abre o modal.

12.1 Estrutura da ficha

A função renderSchoolSheet(row) organiza a ficha em seções:

  1. cabeçalho com tags e nome da escola;
  2. localização e minimapa;
  3. identificação;
  4. oferta educacional;
  5. matrículas por etapa;
  6. contexto de renda;
  7. dados complementares da escola;
  8. contexto de alfabetização.

12.2 Renderização progressiva

Nem tudo da ficha chega de uma vez. Ao abrir o modal, o sistema já renderiza o corpo principal com placeholders. Em seguida, dispara três atualizações assíncronas:

  • updateOpenSchoolModalEnrichment();
  • updateOpenSchoolModalIncome();
  • updateOpenSchoolModalLiteracy().

Isso faz com que o modal abra rápido, mesmo que as integrações complementares demorem.

12.3 Segundo modal: fontes da base

O projeto também ganhou um modal separado para explicar a origem de cada dado usado. Esse modal é aberto pelo link “Verificar fontes da base de dados” no final da home e é montado pela função renderDataSourcesSheet().

12.4 Controle de abertura e fechamento

Para os modais coexistirem sem conflito, o projeto usa:

  • classes is-visible para mostrar e esconder;
  • aria-hidden para acessibilidade;
  • syncModalBodyState() para travar o scroll do body quando qualquer modal estiver aberto;
  • fechamento por botão, clique no overlay e tecla Escape.

13. Cache complementar do Cultura Educa

O projeto não consulta diretamente a ficha pública do Cultura Educa a cada clique. Em vez disso, usa um cache local. Essa decisão melhora a estabilidade, reduz dependência de rede e evita problemas de CORS ou mudanças repentinas da página.

13.1 O que faz o script build-school-enrichment.mjs

  1. lê a base principal para descobrir códigos INEP;
  2. acessa a ficha pública da escola;
  3. extrai tabelas HTML por título;
  4. normaliza linhas e valores;
  5. gera um JSON local por código INEP.

13.2 Ponto importante: parser de booleanos

Um dos problemas reais que surgiu foi a leitura errada de checks visuais. Na fonte pública, havia casos em que o “não” aparecia com um check discreto, e o parser precisava diferenciar isso do check preto forte que significa “sim”.

Esse tipo de problema ensina uma lição importante: scraping de HTML nunca deve assumir que aparência visual e valor lógico são equivalentes sem tratamento explícito.

Lição prática

Sempre que você trouxer um dado externo e ele parecer contraditório, a primeira pergunta deve ser: “o erro está no meu parser, no meu mapeamento ou na fonte original?”.

14. Receitas de manutenção

14.1 Como adicionar um novo filtro simples vindo do JSON

  1. Adicione o select no index.html.
  2. Inclua o novo elemento no objeto els.
  3. Garanta que detectColumns() saiba encontrar a coluna.
  4. Inclua o filtro em FILTER_SELECTS.
  5. Se quiser uma ordenação específica, adicione em FILTER_VALUE_ORDER.
  6. Se ele for um filtro comum, o próprio buildFilters() já vai preencher as opções.

14.2 Como adicionar um filtro derivado

Se o valor não existe como coluna real na base, você precisa tratar em getFilterValue(). Foi exatamente assim que o filtro por região foi resolvido. 

function getFilterValue(row, key) {
  if (key === "region") {
    return getRegionByUf(get(row, "uf"));
  }

  return get(row, key);
}

14.3 Como adicionar um novo campo na ficha da escola

  1. Descubra se o dado vem da base local ou de uma integração externa.
  2. Se vier da base local, garanta o mapeamento em detectColumns().
  3. Monte o campo com renderProfileField() ou uma função de render própria.
  4. Insira o bloco dentro de renderSchoolSheet().
  5. Se depender de dados externos, crie também placeholder e atualização assíncrona.

14.4 Como adicionar um novo painel vindo de API

  1. Declare constantes de tabela, variável, ano e rótulo da fonte.
  2. Crie uma função ensureBaseXLoaded() para buscar e armazenar os dados no estado global.
  3. Crie um snapshot de contexto, como buildIncomeSnapshot() ou buildLiteracySnapshot().
  4. Crie a função de renderização do painel.
  5. Chame a atualização dentro de applyFilters().

14.5 Como ajustar algo só no mobile

  1. Encontre a classe principal do bloco.
  2. Procure os media queries do final do CSS.
  3. Veja se já existe uma regra específica para aquele bloco em @media (max-width: 700px) ou @media (max-width: 560px).
  4. Faça o ajuste ali, não no estilo base, se a mudança for apenas mobile.

14.6 Como pensar novas implementações sem bagunçar tudo

Use este raciocínio:

  1. o novo dado é local, derivado ou externo?
  2. ele entra como filtro, card, painel, modal ou detalhe da ficha?
  3. ele precisa estar disponível logo na abertura da página ou pode ser carregado depois?
  4. ele precisa de cache?

15. Checklist de depuração

Quando alguma coisa quebra, siga este checklist. Ele foi pensado para evitar que você fique “caçando bug no escuro”.

15.1 Se um filtro não funciona

  • confirme se o elemento existe no HTML;
  • confirme se ele está em els;
  • veja se está em FILTER_SELECTS;
  • veja se a coluna existe em detectColumns() ou em getFilterValue();
  • confirme se applyFilters() está sendo chamado após a mudança.

15.2 Se um dado do modal aparece vazio

  • o dado vem do JSON local?
  • o dado vem do cache complementar?
  • a função de render foi chamada antes da resposta assíncrona?
  • o código INEP está correto?
  • houve correspondência automática de município/UF no IBGE?

15.3 Se o texto aparece com acentuação estranha

  • confirme se o arquivo está salvo em UTF-8;
  • confirme se o HTML mantém <meta charset="UTF-8">;
  • veja se o dado já veio corrompido da fonte externa;
  • evite salvar os arquivos em ANSI no editor.

15.4 Se a interface quebra no smartphone

  • veja o bloco no CSS base;
  • depois veja os media queries;
  • procure por max-width, grid-template-columns e width fixo;
  • inspecione se o problema vem de texto sem quebra ou de coluna com largura rígida.

16. Dívidas técnicas e pontos de atenção

Todo projeto real tem compromissos e atalhos. Conhecê-los ajuda muito mais do que fingir que eles não existem.

16.1 O script.js está monolítico

Hoje ele concentra praticamente toda a aplicação. Isso funciona, mas dificulta manutenção futura. Uma evolução natural seria separar em módulos, por exemplo:

  • data/local-base.js;
  • data/ibge.js;
  • ui/filters.js;
  • ui/table.js;
  • ui/modals.js.

16.2 Há lógica legada no final do arquivo

No fim do script.js existem funções e objetos legados ligados à lógica antiga de macroregião e escolas fechadas. Hoje o fluxo principal já substitui isso. Esse trecho é um ótimo candidato a limpeza.

16.3 Há sinais de mojibake em partes do código-fonte

Mesmo quando a interface está sendo corrigida visualmente, o arquivo ainda tem trechos com acentuação corrompida no código. Isso não impede o funcionamento, mas é um sinal de que seria saudável fazer uma revisão geral de codificação e normalização textual.

16.4 Não há testes automatizados

Hoje a validação é majoritariamente manual. Isso é comum em projetos pequenos, mas se o sistema crescer será importante adicionar, pelo menos, testes de funções puras e algumas verificações de regressão em filtros e renderização.

17. Glossário para iniciantes

TermoExplicação simples
Estado globalObjeto que guarda informações importantes do sistema para várias funções reutilizarem.
RenderizarTransformar dados em HTML visível na tela.
SnapshotRetrato do contexto atual, usado para montar cards e painéis.
PlaceholderConteúdo temporário, como “Carregando...”, mostrado antes do dado real chegar.
Fetch assíncronoRequisição de dados que não retorna instantaneamente; o código precisa esperar.
CacheDado armazenado localmente para evitar baixar ou recalcular a mesma coisa várias vezes.
Função derivadaFunção que calcula uma informação a partir de outra, como região a partir da UF.
Delegação de eventoTécnica em que um container escuta cliques de vários elementos internos.
DebounceAtraso controlado para evitar disparar uma ação a cada tecla digitada.
MojibakeTexto com acentuação quebrada por problema de codificação.

18. Como regenerar o PDF

Junto desta documentação existe um script para exportar a versão em PDF a partir deste HTML. Isso é útil porque, sempre que a documentação for atualizada, você consegue gerar um PDF novo sem depender de ferramenta externa manual. 

powershell -ExecutionPolicy Bypass -File .\scripts\export-project-documentation-pdf.ps1

O script usa o Chrome instalado na máquina em modo headless para imprimir este arquivo HTML em PDF.
